## 4. App version and build number

| Field | Value | Status |
|-------|-------|--------|
| `app.json` → `version` | `"0.2.0"` | OK |
| `app.json` → `ios.buildNumber` | `"1"` | ⚠️ **ISSUE** |
| `eas.json` → `production.autoIncrement` | `true` | OK |
| `eas.json` → `cli.appVersionSource` | `"remote"` | OK |

**Issue:** `buildNumber: "1"` in `app.json` is the static fallback. Since `eas.json` has `appVersionSource: "remote"` and `autoIncrement: true`, EAS Build will manage the build number server-side. This is correct **if you've already submitted builds via EAS** — it will auto-increment from the last remote value. If this is the very first EAS build, it'll use `"1"` which is fine.

**No action needed** if using EAS Build. If building locally via Xcode, you must manually bump `buildNumber`.

**Issues found:**

1. **`updates.url` is empty string** (line ~53):
   ```json
   "updates": { "url": "" }
   ```
   If you're not using EAS Update, **remove the entire `updates` block**. An empty URL will cause warnings or errors if the Expo Updates module tries to check for updates at runtime.

2. **`runtimeVersion.policy: "appVersion"`** — This only makes sense with EAS Update. If you're not using OTA updates, remove both `runtimeVersion` and `updates` blocks.

3. **No `ios.privacyManifests`** — Apple now requires privacy manifests for apps submitted to the App Store. Expo SDK 54+ should handle the base manifest, but if you use any APIs that require privacy nutrition labels (e.g., UserDefaults via AsyncStorage), you may need to declare them. Check Apple's latest requirements.

**One non-reactive Zustand usage (not dead code, but a bug):**

- **`App.tsx:140`** — `useAppStore.getState().workspace?.name` called inside JSX render:
  ```tsx
  headerTitle: useAppStore.getState().workspace?.name || 'Copilot Chat',
  ```
  `getState()` reads the store **once** and is not reactive — the header title won't update when `workspace` changes (e.g., after connection). This should use a reactive selector instead.
